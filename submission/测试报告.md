## 4. Bug 汇总

### 4.1 测试过程中发现的缺陷

| Bug ID | 标题 | 严重程度 | 根因 | 状态 |
|--------|------|----------|------|------|
| BUG-T01 | 测试与生产环境数据库冲突 | 🟡 中 | conftest 未检测端口占用，测试和生产共用同一 DB 文件 | ✅ 已修复 |
| BUG-T02 | 生产服务运行时测试失败 | 🟡 中 | SQLite 写锁冲突导致 `attempt to write readonly database` | ✅ 已修复 |

### 4.2 BUG-T01 详细分析

**发现场景**：在已有生产后端服务运行的情况下执行 `pytest`，2 个答题提交测试失败（HTTP 500）。

**根因**：
```
1. conftest 删除 math_practice.db
2. conftest 启动新服务在 8000 端口，但端口已被生产服务占用
3. 新服务启动失败（端口冲突）
4. 测试请求实际打到生产服务
5. 生产服务的 DB 已被删除，重新初始化但权限异常
6. SQLite 报错 "attempt to write a readonly database"
```

**影响范围**：test_submit_correct_answer、test_submit_wrong_answer 两用例失败，31/33 通过。

**修复方案**：
```python
# conftest.py 增加端口检测逻辑
def _is_port_in_use(port):
    with socket.socket(socket.AF_INET, socket.SOCK_STREAM) as s:
        return s.connect_ex(('localhost', port)) == 0

# 若端口已被占用，先杀掉旧进程
if _is_port_in_use(8000):
    subprocess.run(["pkill", "-f", "uvicorn main:app"])
    time.sleep(1)
```

**修复后验证**：33/33 全部通过，测试时间 3.56 秒。

## 5. 典型 Bug 分析

### 5.1 SQLite 并发写入冲突

**严重程度**：🟡 中等  
**影响**：测试与生产服务并存时测试失败  
**根因深度分析**：

SQLite 是文件级锁数据库，不支持并发写操作。当同一个 `math_practice.db` 文件被两个 Python 进程同时打开时：
1. 第一个进程持有写锁
2. 第二个进程尝试写入时触发 SQLite `SQLITE_BUSY` 错误
3. SQLAlchemy 将其包装为 `OperationalError`

**改进方向**：
- 测试使用独立数据库文件（如 `test_math_practice.db`）
- 或使用 SQLite `:memory:` 内存数据库进行测试
- 或使用工厂模式创建数据库引擎，支持配置切换
