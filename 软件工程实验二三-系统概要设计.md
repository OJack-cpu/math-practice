# 软件工程实验二三：系统概要设计

## 实验信息

| 项目 | 内容 |
|------|------|
| 实验名称 | 系统概要设计 |
| 实验序号 | 实验二 + 实验三（合并） |
| 课时 | 4 学时 |
| 对应阶段 | 软件工程生命周期 — 设计阶段（概要设计） |

## 实验目的

1. 掌握从需求到系统架构的映射方法
2. 学会通过 AI Agent 辅助进行系统设计
3. 能够用可视化方式表达设计决策
4. 了解不同设计方案的 trade-off 分析

## 实验工具

| 工具 | 用途 | 方式 |
|------|------|------|
| OpenCode | AI Agent，核心交互环境 | 已安装 |
| drawio-skill + Draw.io MCP | 架构图、ER图、流程图、线框图 | 通过 OpenCode 安装 |
| excalidraw-skill | 手绘风格原型、UI 草图 | 通过 OpenCode 安装 |

## 实验前置条件

1. 已完成实验一（需求获取与分析），需求规格说明书已定稿
2. 已在终端中配置好 OpenCode
3. 已安装至少一个绘图 Skill（见下方说明）

### Skill + MCP 安装

在 OpenCode 终端中，通过自然语言安装所需的绘图能力：

```
你：帮我安装 drawio-skill，我要画架构图和设计图
```

OpenCode 会自动完成安装。你也可以通过以下命令手动安装：

```
# 方法一：npx skills 安装
npx skills add Agents365-ai/365-skills -g

# 方法二：克隆到 OpenCode skill 目录
git clone https://github.com/Agents365-ai/drawio-skill.git ~/.config/opencode/skills/drawio-skill
```

装好后，在 OpenCode 中直接说：

```
帮我画一个系统架构图，用 Draw.io，系统分为前端、后端、数据库三层
```

AI 会调用 Skill 的绘图知识 + MCP 的执行能力，直接生成并在浏览器中打开。

## 实验内容

本实验对应软件工程中"概要设计"阶段。你需要在实验一的需求基础上，对系统进行整体的架构设计。

```
需求规格说明书 → 概要设计 → 详细设计（下次实验）
                                  ↓
                             编码实现
```

## 实验步骤

### 第 1 步：需求回顾与补充（课堂内）

**目标**：确保需求文档完整，为设计打下基础。

1. 打开实验一提交的需求规格说明书
2. 与 Agent 进行一次设计准备对话：

```
与 Agent 对话示例：

你：这是我之前的需求文档。[粘贴需求文档]
    我现在要开始做系统设计了。
    请帮我检查：
    1. 需求文档中是否有模糊、不完整的地方需要先补充？
    2. 从设计角度看，这个系统最核心的设计难点是什么？
    3. 根据这个项目的类型，我应该在概要设计中重点设计哪些方面？
```

3. 根据 Agent 的建议，补充完善需求文档中缺失的部分

### 第 2 步：技术选型与架构方案对比

**目标**：确定技术栈和整体架构方案。

```
与 Agent 对话示例：

你：请为我的项目推荐 2~3 个技术方案，从以下维度对比：
    1. 技术栈（前端、后端、数据库、第三方服务）
    2. 架构模式（前后端分离？单体？微服务？）
    3. 各方案的优缺点和适用场景
    4. 我的推荐方案及理由

    我的项目简介：[简要描述项目]
    主要约束：[技术约束、时间约束等]
```

### 第 3 步：系统设计（核心环节）

**目标**：产出概要设计的核心内容。

从以下方向中选择 **3~4 个**，适合你项目的即可。如果你不确定选哪些，问问 Agent：

```
"根据我的项目类型，概要设计阶段我该重点做哪些方面？"
```

#### 内容池（选择 3~4 个）

| 方向 | 说明 | 推荐工具 |
|------|------|---------|
| **系统架构图** | 整体分层、模块划分、服务间关系、部署方式 | drawio-skill |
| **数据模型 / ER 图** | 核心实体关系、数据结构、主要字段 | drawio-skill / mermaid-skill |
| **界面原型 / 线框图** | 核心页面的布局草图、用户操作流程 | excalidraw-skill / drawio-skill |
| **接口概览** | 模块间或系统间的主要接口、通信方式 | drawio-skill / mermaid-skill |
| **核心流程设计** | 关键业务逻辑的流程图（如有） | drawio-skill / mermaid-skill |

每个方向的对话示例：

```
你：请帮我画这个系统的架构图。
    系统采用前后端分离架构。
    前端用 [你的技术栈]。
    后端用 [你的技术栈]。
    数据库用 [你的选择]。
    外部服务有 [如有]。
    用 Draw.io，保存到 docs/architecture.drawio。
```

```
你：请帮我画 ER 图。
    核心实体有：用户、[你的实体]。
    它们之间的关系是：[描述关系]。
    用 Draw.io 或 Mermaid，保存到 docs/er-diagram。[文件格式]
```

```
你：请帮我设计 [页面名] 的界面原型。
    这个页面包含：[列出主要元素]。
    用户操作流程：[描述]。
    用 Excalidraw 或 Draw.io，保存到 docs/[文件名]。[文件格式]
```

### 第 4 步：设计说明与决策记录

**目标**：用文字解释你的设计决策和 trade-off。

每个设计图需要附一段设计说明（100~300 字），包含：

- 这张图表达了什么
- 为什么这样设计（有哪些方案被放弃）
- 这个设计有什么局限性或后续可以改进的地方

### 第 5 步：设计评审

**目标**：以第三方视角审查设计方案的合理性。

```
与 Agent 对话示例：

你：请以第三方评审者的视角，审查我的概要设计。
    重点检查：
    1. 架构设计是否合理？有没有过度设计或设计不足？
    2. 数据模型是否覆盖了需求中的所有实体？
    3. 技术选型是否适合这个项目？
    4. 设计方案之间是否一致？（比如 API 设计和 UI 是否对应）
    5. 有没有遗漏重要的设计决策？

    [粘贴你的设计文档]
```

根据评审意见修改设计，并记录发现了几个问题、修改了什么。

## 评分标准

| 评分项 | 分值 | 评分要点 |
|--------|------|---------|
| **技术选型** | 20 分 | 方案对比完整、选型理由充分 |
| **架构设计** | 25 分 | 架构图清晰、模块划分合理、设计说明完整 |
| **数据/界面/接口设计** | 25 分 | 覆盖核心需求、设计合理、图文并茂 |
| **设计说明与决策分析** | 10 分 | 能解释为什么这样设计、有 trade-off 分析 |
| **设计评审** | 10 分 | 发现问题的数量和质量、修改的合理性 |
| **文档规范与工具使用** | 10 分 | 格式规范、工具使用恰当、Skill/MCP 配置合理 |

## 实验提示

### 不同项目类型的设计重点

| 项目类型 | 建议重点设计的方面 |
|---------|------------------|
| 前后端 Web 应用 | 系统架构图 + ER 图 + 界面原型 + API 概览 |
| AI / 数据处理应用 | 系统架构图 + 核心流程设计 + 数据模型 |
| 移动端应用 | 界面原型（重点）+ 系统架构 + 接口概览 |
| 工具类/算法类 | 核心流程设计 + 架构图 + 数据模型 |
| 游戏/创意应用 | 界面原型 + 系统架构 + 数据设计 |

### Skill 切换技巧

在 OpenCode 中，你可以随时切换使用的 Skill：

```
用 drawio-skill 画架构图
用 excalidraw-skill 画 UI 原型
用 mermaid-skill 画流程图
```

如果不指定，AI 会根据你的描述自动选择最合适的工具。

### 常见错误

| 错误 | 说明 |
|------|------|
| 设计未覆盖所有需求 | 检查每个功能模块是否都有对应的设计 |
| 过度设计 | 只设计 MVP 范围内需要的部分，不要为未来可能不存在的功能做设计 |
| 技术选型理由不足 | 不要只说"我用 React 因为流行"，要说明它为什么适合这个项目 |
| 设计图之间不一致 | 架构图中的模块、ER 图中的实体、界面中的功能要对得上 |
