Section 02

实验项目选题

选择一个贯穿 8 次实验的项目，后续所有实验均围绕同一项目展开

选题原则

  • 1 自己会用 — 项目做出来你自己愿意继续使用，不是交完作业就删
  • 2 有技术深度 — 不是纯 CRUD，有明确的技术难点需要攻克
  • 3 能走完 SE 流程 — 需求、设计、编码、测试、部署全链路
  • 4 小而美 — 功能聚焦，不追求大而全，追求体验精致

项目难度分级

A 档（基础）
功能明确，技术栈简单，2 小时内 Agent 能出可运行版本

适合：基础较弱、想稳拿分的同学

B 档（进阶）
有业务逻辑复杂度，需要多技术整合

适合：有一定基础、想拿高分的同学

C 档（挑战）
技术整合复杂度高，有明确的工程难点

适合：基础扎实、想冲刺加分的同学

评分说明：A/B/C 档均可拿到同等基础分。选择 B/C 档且完成质量高的，可获得技术复杂度加分。不建议盲目选难——选自己能做完、能做好的。

参考选题

1

考研/考公刷题系统

★★☆

智能错题本 + 间隔复习 + 薄弱点分析 + 学习数据可视化

间隔重复算法
实现 SM-2 算法（或 FSRS 算法），根据答题正确率、回忆难度动态计算下次复习时间
题库数据模型
题目表、答题记录表、复习调度表、知识点标签表；多表关联，状态机设计
数据可视化
ECharts 或 Chart.js：每日学习量折线图、知识点掌握度雷达图、正确率趋势图
薄弱点分析
按知识点聚合正确率，生成薄弱报告
前端交互
Vue/React + 动效：答题翻牌效果、进度条动画
部署
前后端分离架构，前端静态部署 + 后端 API 服务
2

AI 面试陪练

★★★

AI 模拟技术面试官，支持语音对话 + 数字人形象 + 实时评分

语音识别（STT）
OpenAI Whisper API 或 faster-whisper；流式识别、降噪处理、中英文混合识别
语音合成（TTS）
edge-tts（免费）或 Azure TTS；多音色选择、语速调节、流式播放
数字人驱动
D-ID API / HeyGen API（商业方案）或 SadTalker（开源）；音频驱动口型同步
面试对话引擎
Prompt Engineering：角色扮演 + 多轮上下文管理；系统提示词设计
实时评分
每轮回答后 AI 评分（技术深度、表达清晰度、逻辑性）；评分维度设计
对话记录
完整对话历史存储 + 复盘功能；音频存储、文字转录、回放

个人 AI Agent

★★★

一个有记忆、能调用工具的私人 Agent

Agent 架构
ReAct 模式（Reasoning + Acting）：用户意图 → 选择工具 → 执行 → 返回结果
对话管理
多轮对话上下文维护、对话历史压缩；长对话 token 超限的处理策略
记忆系统
短期记忆（当前对话）+ 长期记忆（跨会话）；向量数据库存储、相似度检索
工具调用
Function Calling / Tool Use；实现 3-5 个工具：天气查询、计算、网页搜索等
流式输出
SSE（Server-Sent Events）；逐字输出效果，前端打字机动画
本地知识库
上传 PDF/文档 → Agent 能检索回答；文档解析、向量化、RAG 检索

AI 论文阅读伴侣

AI 记账助手

Section 04

实验步骤

按照以下 5 个步骤完成本次实验

1

选题确认与初步描述

⏱ 20 分钟

目标：明确你要做什么，并用自然语言向 Agent 描述。

操作

  • 确定你的项目选题（从参考列表中选，或自拟）
  • 向 Agent 描述你的项目，包含以下要素
对话示例 
你：我要做一个 [项目名称]。
    目标用户是 [谁]。
    核心功能是 [做什么]。
    解决的问题是 [为什么需要]。
    技术栈初步想法是 [前端/后端/数据库]。
  • 与 Agent 进行 2-3 轮对话，让 Agent 帮你梳理初步需求

要求

  • ! 对话记录截图或文本，后续需要提交
  • ! 选题需获得教师确认后方可进入下一步
2

需求获取 — 与 Agent 深度对话

⏱ 30 分钟

目标：通过与 Agent 的多轮对话，挖掘出完整的需求列表。

按以下 4 个维度，分批向 Agent 提问：

2.1 功能需求

你：请帮我分析 [项目名称] 需要哪些功能模块。
    我初步想到的有：[列出你想到的功能]。
    请帮我检查有没有遗漏，并给出功能模块的层级划分。

追问方向：

  • "除了这些，用户还会有什么期望但没说出来的需求？"
  • "这个功能可以拆分成哪些子功能？"
  • "有没有 MVP（最小可行产品）版本的功能子集？"

2.2 用户角色

你：这个系统有哪些类型的用户？
    每种用户的核心操作是什么？
    不同角色之间的权限有什么区别？

2.3 非功能需求

你：这个系统有哪些非功能需求？
    请从性能、安全、可用性、兼容性等维度帮我分析。

追问方向：

  • "并发量大概多少？需要支持多少用户同时在线？"
  • "数据安全有什么要求？需要加密吗？"
  • "需要支持移动端吗？响应式设计？"

2.4 约束条件

你：这个项目有哪些技术约束和业务约束？
    比如必须用什么技术栈、不能用什么、有什么合规要求？

产出

与 Agent 的完整对话记录（文本）

3

需求整理 — 功能分解

⏱ 20 分钟

目标：将对话中获取的需求整理为结构化的功能分解图。

操作

  • 将第 2 步的对话内容喂给 Agent
你：请根据我们的对话，帮我生成这个系统的功能分解图。
    用 Mermaid flowchart 语法输出。
    格式：系统 → 模块 → 子功能，共三层。
  • Agent 会生成 Mermaid 代码，复制到 Mermaid Live Editor 渲染查看
  • 检查并让 Agent 修改，直到满意

示例输出（以考研刷题系统为例）

flowchart TD
    A[考研刷题系统] --> B[题库管理]
    A --> C[刷题练习]
    A --> D[复习调度]
    A --> E[数据分析]
    A --> F[用户管理]

    B --> B1[题目录入]
    B --> B2[题目分类]
    B --> B3[题目搜索]

    C --> C1[顺序练习]
    C --> C2[随机练习]
    C --> C3[错题重练]

    D --> D1[SM-2 算法调度]
    D --> D2[每日复习计划]
    D --> D3[复习提醒]

    E --> E1[正确率统计]
    E --> E2[薄弱知识点]
    E --> E3[学习时长]

    F --> F1[注册登录]
    F --> F2[个人信息]

要求

  • 1 功能分解至少 2 层（系统 → 模块 → 子功能）
  • 2 每个模块至少 2-3 个子功能
  • 3 标注哪些是 MVP 功能（用颜色或文字标记）
4

需求整理 — 用例建模

⏱ 20 分钟

目标：识别系统的核心用例，绘制用例图并编写用例描述。

4.1 识别参与者和用例

你：请帮我识别 [项目名称] 的参与者和核心用例。
    用例图用 Mermaid 语法输出。
    每个用例需要标注参与者。
    用例之间如果有包含/扩展关系也要标注。

4.2 编写核心用例描述

选择 3 个核心用例，用以下格式编写：

你：请为以下 3 个核心用例编写详细描述：
    1. [用例名称]
    2. [用例名称]
    3. [用例名称]
    
    格式要求：
    - 用例编号
    - 用例名称
    - 参与者
    - 前置条件
    - 基本流程（主成功场景）
    - 扩展流程（异常/备选场景）
    - 后置条件

用例描述模板

字段 内容
用例编号 UC-001
用例名称 用户登录
参与者 普通用户
前置条件 用户已注册账号
基本流程 1. 用户输入账号密码 → 2. 系统验证 → 3. 登录成功，跳转主页
扩展流程 3a. 密码错误 → 提示重试；3b. 连续 5 次错误 → 锁定账号
后置条件 用户会话建立

要求

  • 1 用例图用 Mermaid 语法输出
  • 2 核心用例至少 3 个，每个用例的描述不少于 15 行
  • 3 包含至少 1 个异常流程
5

Section 07

实验提示

掌握 Prompt 技巧，避免常见错误

Prompt 技巧

01

给 Agent 足够上下文

不要只说"帮我做需求分析"，要告诉它你的选题、用户、场景

02

分步提问

一次问一个维度（功能、用户、性能），不要一次扔所有问题

03

追问

Agent 第一次回答往往不够深入，要追问"还有吗？""为什么？""怎么验证？"

04

要求格式化输出

明确告诉 Agent 输出 Mermaid 代码、表格、还是列表

常见错误

错误 正确做法
✗ 需求太模糊："用户能管理数据" ✓ 具体化："用户可以增删改查题目，支持按知识点筛选"
✗ 非功能需求不可验证："系统要快" ✓ 可验证："页面加载时间 < 2 秒，API 响应时间 < 500ms"
✗ 用例粒度不对：一个用例包含太多操作 ✓ 拆分：每个用例只描述一个完整的业务动作
✗ 忽略异常流程：只写主成功场景 ✓ 补充：密码错误、网络超时、数据冲突等异常场景
